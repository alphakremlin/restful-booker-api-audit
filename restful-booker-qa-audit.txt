EVIDENCE COLLECTOR — BUG REPORT
---------------------------------

BUG #1 — Wrong HTTP Status on Failed Authentication
  Endpoint : POST /auth
  Payload  : {"username":"wrong","password":"wrong"}
  Expected : 401 Unauthorized
  Actual   : 200 OK  {"reason":"Bad credentials"}
  Severity : HIGH
  Detail   : Any client checking HTTP status code to determine auth
             success will treat a failed login as successful. Breaks
             standard auth flows and automated test assertions.
  Fix      : Return 401 Unauthorized when credentials are invalid.

BUG #2 — Resource Creation Returns 200 Instead of 201
  Endpoint : POST /booking
  Expected : 201 Created
  Actual   : 200 OK
  Severity : MEDIUM
  Detail   : Violates REST conventions. Clients and API gateways rely
             on 201 to confirm resource creation. Causes failures in
             automated pipelines that assert correct status codes.
  Fix      : Return 201 Created on successful booking creation.

BUG #3 — Missing Required Fields Causes 500 Server Error
  Endpoint : POST /booking
  Payload  : {"firstname":"John"}
  Expected : 400 Bad Request with field-level validation errors
  Actual   : 500 Internal Server Error
  Severity : CRITICAL
  Detail   : Server crashes on malformed input. In production this
             exposes internal error state, leaks stack traces, and
             signals exploitable input paths to attackers.
  Fix      : Add input validation. Return 400 with descriptive
             error message listing missing required fields.

BUG #4 — Invalid Date Filter Crashes Server
  Endpoint : GET /booking?checkin=not-a-date
  Expected : 400 Bad Request
  Actual   : 500 Internal Server Error
  Severity : CRITICAL
  Detail   : Unvalidated query parameter crashes the endpoint.
             Any malformed date input in a query string takes down
             the booking search functionality entirely.
  Fix      : Validate date format before processing. Return 400
             with message: "Invalid date format. Use YYYY-MM-DD."

BUG #5 — Negative Price Accepted on Booking Creation
  Endpoint : POST /booking
  Payload  : {"totalprice": -999, ...}
  Expected : 400 Bad Request
  Actual   : 200 OK — booking created with totalprice: -999
  Severity : HIGH
  Detail   : No numeric validation on totalprice field. In a real
             system this corrupts financial records and could be
             exploited to create negative-value transactions.
  Fix      : Validate totalprice is a positive number >= 0.

BUG #6 — Auth Endpoint Accepts Incomplete Payload Silently
  Endpoint : POST /auth
  Payload  : {"username":"admin"}  (password field omitted)
  Expected : 400 Bad Request
  Actual   : 200 OK  {"reason":"Bad credentials"}
  Severity : MEDIUM
  Detail   : Incomplete payloads are silently processed rather than
             rejected. Indicates missing schema validation on the
             auth request body.
  Fix      : Validate that both username and password fields are
             present. Return 400 if either is missing.

BUG #7 — Security Headers Missing / Tech Stack Exposed
  Observed : X-Powered-By: Express (present in all responses)
  Missing  : X-Content-Type-Options: nosniff
             X-Frame-Options: DENY
             Strict-Transport-Security: max-age=31536000
             Content-Security-Policy
  Severity : MEDIUM
  Detail   : X-Powered-By reveals the server technology to attackers.
             Missing security headers are flagged by OWASP ZAP,
             Burp Suite, and all standard security scanners.
  Fix      : Add helmet.js middleware to Express to set all
             security headers and remove X-Powered-By automatically.
             npm install helmet
